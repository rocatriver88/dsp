# 评审整改任务单 V2（2026-04-14）

## 说明
本版基于 V1 以及 Claude Code 的复核意见整理。结论是：

- **接受**：creative 相关越权风险补充、TDD 先行、`admin-secret` 默认值删除、`api_key` 默认隐藏、P2 聚焦端到端回归。
- **部分接受并修正**：win/impression 迁移方案、shutdown 顺序、`Validate()` 的实现形式。
- **不接受原样执行**：在 bidder 仍双写 `win + impression` 时，直接把报表聚合改成 `event_type IN ('win','impression')`。这会把当前同一次赢标双记为两次曝光，结果比现状更糟。

本文档的目标不是规定唯一实现手法，而是给出**必须达成的行为目标、可接受的迁移顺序和验收标准**。

## 决策摘要

### 接受的修正
1. creative handler 的 ownership 校验必须纳入 P0，同批修。
2. `admin-secret` 默认值应删除，不保留任何环境 fallback。
3. `Advertiser.APIKey` 当前优先用 `json:"-"` 隐藏；如果后续确实出现“管理员可见、广告主不可见”的双形态，再引入 DTO。
4. 测试应并入各整改项，不要把关键安全测试推迟到最后统一补。
5. P2 应以跨 handler、跨进程的端到端回归为主，而不是重复 P0/P1 已覆盖的单点单测。

### 修正后接受的点
1. **win/impression 需要分步迁移**，但 Step A 不能用简单 OR 聚合，必须先引入**去重后的有效曝光口径**，兼容当前双写和未来单写。
2. **shutdown 需要顺序约束**，但文档只规定行为依赖关系，不强行绑定某一种具体代码结构。
3. `Config.Validate()` 的目标是“启动期强制失败”，不强制要求一定改成 `error return`；保留 `log.Fatal` 也可以，只要调用点真的执行到了。

### 本版新增修正
1. `HandleListCreatives` 也存在越权读取问题，不能只修 create/update/delete。
2. creative CRUD 修复时，建议同步补 `campaign:updates` 发布，和现有 [biz-engine.md](/C:/Users/Roc/github/dsp/docs/contracts/biz-engine.md) 契约保持一致。
3. 越权错误码允许使用 `403` 或 `404`，但必须**统一且稳定**，不能同类接口一会儿 403 一会儿 404。

## 执行原则
- 先安全边界，后数据语义，最后运行时治理。
- 每个整改项先补测试再修代码；至少覆盖一个正向场景和一个反向场景。
- 不把本轮整改扩展成大重构；能用局部修复解决的问题，不顺手做架构翻新。
- 只对必要接口做 API 形状修改；如果改了路由或 schema，记得同步 `make api-gen`。

## P0：租户隔离与敏感信息泄露

### 问题范围
- advertiser 读取未强制本人 scope。
- billing 接口信任调用方传入 advertiser ID。
- report 接口大多只按 campaign ID 查报表，没有 owner check。
- creative 相关接口缺 ownership 校验，包含读和写。
- advertiser 对象会直接序列化 `api_key`。
- 前端存在 `advertiser_id = 1` 的硬编码调用。

### 必做改动
1. advertiser 接口
- `GET /api/v1/advertisers/{id}` 只能返回当前认证广告主本人数据。
- `Advertiser.APIKey` 改为默认不序列化，当前推荐直接用 `json:"-"`。

2. billing 接口
- `POST /api/v1/billing/topup` 忽略请求体中的 `advertiser_id`，强制用认证上下文。
- `GET /api/v1/billing/transactions` 不再允许查询任意 advertiser。
- `GET /api/v1/billing/balance/{id}` 必须做 owner check；如果要顺手收敛成 `/billing/balance` 也可以，但不是必须。

3. reports 接口
- 以下接口统一在进入 reporting 查询前先校验 campaign 归属：
  - `/api/v1/reports/campaign/{id}/stats`
  - `/api/v1/reports/campaign/{id}/hourly`
  - `/api/v1/reports/campaign/{id}/geo`
  - `/api/v1/reports/campaign/{id}/bids`
  - `/api/v1/reports/campaign/{id}/attribution`
- `simulate` 和 `export` 已有 owner check，保持一致。

4. creative 接口
- `GET /api/v1/campaigns/{id}/creatives` 先校验该 campaign 属于当前 advertiser。
- `POST /api/v1/creatives` 先校验 `campaign_id` 归属。
- `PUT /api/v1/creatives/{id}`、`DELETE /api/v1/creatives/{id}` 先定位 creative，再校验其所属 campaign。
- creative CRUD 修复时，同步补上 `campaign:updates` publish，符合 [biz-engine.md](/C:/Users/Roc/github/dsp/docs/contracts/biz-engine.md)。

5. 前端调用
- 移除 [page.tsx](/C:/Users/Roc/github/dsp/web/app/billing/page.tsx) 中对 advertiser `1` 的硬编码。
- `web/lib/api.ts` 的 billing 相关函数改成默认由后端按 auth context 决定 advertiser。

### 测试要求
- A 的 key 访问 B 的 advertiser/balance/transactions/report/creative 全部失败。
- A 的 key 不能给 B 充值，也不能把 creative 创建到 B 的 campaign 下。
- 所有返回 advertiser 的接口响应体都不包含 `api_key`。
- creative 合法更新/删除后，会触发 `campaign:updates`。

## P0：管理面与启动配置安全

### 问题范围
- `Config.Validate()` 存在但未在主进程启动时执行。
- admin 鉴权允许默认 `admin-secret`。
- admin 鉴权接受 query 参数 `admin_token`。

### 必做改动
1. 启动校验
- `cmd/api` 和 `cmd/bidder` 在 `config.Load()` 后立即执行配置校验。
- 目标是：生产缺少关键 secret 时，进程直接失败退出。

2. admin 鉴权
- 删除 `admin-secret` 默认值。
- 删除 `admin_token` query 参数支持，只保留 `X-Admin-Token` header。

3. 默认值策略
- 明确 `BIDDER_HMAC_SECRET`、`ADMIN_TOKEN`、`CORS_ALLOWED_ORIGINS` 哪些仅用于开发。
- 文档和注释要能解释默认值边界。

### 测试要求
- 生产环境缺少 `BIDDER_HMAC_SECRET` 或 `ADMIN_TOKEN` 时启动失败。
- header 正确时 admin 通过，query 带 token 时必须失败。
- 未设置 admin token 时，进程不能靠默认值启动。

## P1：事件语义、计费与报表口径

### 问题范围
- `handleWin` 当前把一次赢标同时记成 `win` 和 `impression`。
- 现有 reporting/autopause 以 `impression` 为曝光口径，和当前伪曝光耦合。
- `handleClick` 没有去重，CPC 可能重复扣费。
- click/convert 事件用 request context 异步发送，handler 返回后可能取消。

### 迁移原则
- **不能**在当前双写状态下，把曝光聚合直接改成 `event_type IN ('win','impression')`。
- 迁移必须让“当前双写”和“未来单写”都不会双计或漏计。

### 建议迁移顺序
1. Step A：先引入“有效曝光”口径
- 在 reporting/autopause 等依赖曝光统计的地方，改为按 `request_id` 去重计算有效曝光，兼容 `win` 与 `impression`。
- 可接受的实现方式包括：
  - `countDistinctIf(request_id, event_type IN ('win','impression'))`
  - 或其他能保证同一次赢标当前只算一次曝光、未来单写也仍正确的等价方案
- 这一阶段 bidder 仍保持现状双写，不先动回调。

2. Step B：再停止伪造 impression
- 删除 `handleWin` 中的 `SendImpression`。
- 保留 `win` 事件，真实曝光若未来引入单独回调，再由独立来源写 `impression`。

3. Step C：修 click/convert 的一致性
- `handleClick` 增加去重，至少对同一 `request_id` 在合理 TTL 内只扣一次费。
- `SendClick`、`SendConversion` 改用独立 context 或带 timeout 的后台 context，不再直接复用 request context。

### 结果要求
- 迁移前后 CTR、曝光、花费指标不能出现人为跳变。
- `win` 不再自动产生一条 `impression` 行。
- 重复 click 不会重复扣费。
- click/convert 在 handler 返回后仍可稳定写入 Kafka 或缓冲。

### 测试要求
- 当前双写数据下，有效曝光不会被重复统计。
- 停止伪造 impression 后，新数据的报表口径保持稳定。
- 同一 `request_id` 连续 click 多次只扣费一次。
- request context 取消后，click/convert 仍能进入 producer/mock buffer。

## P1：生命周期与优雅关闭

### 问题范围
- 主进程用 `context.Background()` 驱动长期 goroutine。
- shutdown 只停 HTTP server，没统一 cancel worker。

### 必做改动
1. 根 context
- API 和 bidder 使用可 cancel 的根 context，例如 `signal.NotifyContext`。
- 长期后台任务都绑定到同一根 context。

2. 关闭约束
- 必须满足以下依赖关系：
  - 先发出根 cancel，让后台 worker 开始停。
  - 再 drain HTTP inflight 请求。
  - 再关闭/flush producer。
  - 最后关闭 DB/Redis/ClickHouse 等底层连接。
- 只要实现满足上述依赖关系，不限定具体代码写法。

3. 失败策略
- 对 Redis、Kafka、ClickHouse 的 fail-open/fail-closed 行为做显式文档化。

### 测试要求
- cancel 根 context 后，各 worker 能在时限内退出。
- 收到退出信号后，服务不会继续产生不受控后台写入。

## P2：端到端回归保护

### 目标
P2 只保留跨 handler、跨进程的回归保护，防止以后新增接口再次漏掉 scope 或 shutdown 约束。

### 必做改动
1. 租户隔离集成测试
- 启完整 api handler tree。
- 创建 A/B 两个 advertiser，穷举 A 访问 B 的 advertiser、billing、campaign、report、creative 路径。
- 统一断言这些越权访问被拒绝。

2. 响应体敏感字段扫描
- 对所有返回 advertiser 的路径，断言 JSON 不包含 `api_key`。

3. shutdown 集成测试
- 启动 api/bidder，触发退出流程，验证 worker 与 inflight 行为符合约束。

4. 前端手工 QA
- billing 页面登录后查看余额、充值、查看交易记录。
- campaign/report 页面验证 scope 正常，无硬编码 advertiser 假设残留。
