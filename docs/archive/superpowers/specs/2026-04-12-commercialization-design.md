# DSP 商业化落地设计

## 概述

将 DSP 系统从开发态推向正式商业使用的分阶段落地方案。

**策略：** 技术验证与合规申请并行推进，分三个阶段：自己操盘验证 → 接入真实流量小额实投 → 邀请代理商内测。

**团队分工：** 技术侧一人负责全部开发和运维，商务/运营合伙人负责合规资质、Exchange 合作、代理商拓展。

**目标用户：** 第一批为 3-5 家已有关系的中小代理商，定向邀请制，不公开注册。

## Phase 1：生产部署 + 自己操盘（第 1-2 月）

### 1.1 部署架构

单台云服务器（4C8G）+ Docker Compose 部署全套服务。

| 组件 | 方案 |
|------|------|
| 服务器 | 阿里云/腾讯云 ECS 4C8G |
| 部署方式 | Docker Compose（复用现有配置） |
| 反向代理 | Nginx + Let's Encrypt HTTPS |
| 数据备份 | PostgreSQL 每日自动备份 + 云盘快照 |

暂不引入 K8s，单人运维开销不合算。等日请求量过百万再考虑。

### 1.2 生产加固清单

| 领域 | 要做的事 | 优先级 |
|------|---------|--------|
| 配置 | 环境变量外部化，敏感信息不进代码 | P0 |
| HTTPS | Nginx + Let's Encrypt，全站强制 HTTPS | P0 |
| 数据备份 | PostgreSQL 每日自动备份 + 云盘快照 | P0 |
| 监控告警 | Prometheus + Grafana 已有；加飞书/钉钉 webhook 告警 | P0 |
| Redis 持久化 | AOF 持久化，防止重启丢预算数据 | P0 |
| 日志 | 结构化日志已有；加日志轮转防磁盘撑爆 | P1 |
| 限流 | API 限流已有；生产环境调整阈值 | P1 |
| 数据库 | 连接池配置、慢查询监控 | P1 |
| 健康检查 | Docker 容器健康检查 + 自动重启 | P1 |

### 1.3 自动化操盘模拟器（autopilot）

新增服务 `cmd/autopilot/`，两个运行模式：

#### 模式一：`autopilot verify` — 全链路验证

一次性运行，按顺序执行以下场景，每步通过无头浏览器截取前端 UI 截图：

| 步骤 | 操作 | 截图内容 |
|------|------|---------|
| 1 | API 创建广告主 | 登录后首页 Dashboard（空状态） |
| 2 | 充值测试余额 | Billing 页面，余额变化 |
| 3 | 创建 Campaign + Creative | Campaign 列表页、详情页 |
| 4 | 启动 Campaign，exchange-sim 发流量 | Analytics 实时面板（数据流入） |
| 5 | 等待积累数据（默认 5 分钟，可配置） | Reports 页面（展示量、点击、花费） |
| 6 | 暂停 Campaign | Campaign 状态变为 paused |
| 7 | 异常注入：预算耗尽 | Campaign 自动暂停、余额归零 |
| 8 | 异常注入：服务重启（重启 bidder 容器） | Grafana 监控面板（中断 + 恢复） |
| 9 | 异常注入：Kafka 延迟（暂停 consumer 后恢复） | Analytics 数据追平过程 |
| 10 | 生成最终报告 | — |

**输出：** HTML 报告，包含：
- 每步 Pass/Fail 状态
- 每步 UI 截图（标注时间戳）
- API 响应摘要（状态码、关键字段）
- 异常场景系统恢复时间
- 数据一致性校验（Redis 花费 vs ClickHouse 花费）

#### 模式二：`autopilot run` — 持续模拟

长驻运行，模拟真实投放节奏：

- **流量曲线：** 白天（8:00-22:00）高流量，夜间低流量
- **操盘行为：** 随机创建/暂停/调整 campaign
- **健康检查：** 每分钟检查各服务状态，异常时发钉钉/飞书告警
- **日报：** 每天 09:00 自动生成前一天运行摘要（含关键 Dashboard 截图）

#### 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| 主程序 | Go（`cmd/autopilot/`） | 和现有代码栈一致，复用 API client |
| 无头浏览器 | chromedp | Go 生态最成熟的 Chrome DevTools Protocol 库 |
| 报告生成 | Go HTML template | 轻量，不引入额外依赖 |
| 异常注入 | Docker API | 通过 Docker socket 重启/暂停容器 |
| 告警通知 | Webhook（钉钉/飞书） | 商务合伙人也能收到 |

## Phase 2：接入真实 Exchange + 小额实投（第 2-3 月）

### 2.1 Exchange 对接适配层

```
internal/exchange/
├── adapter.go          # Exchange 接口定义
├── sim_adapter.go      # 现有模拟器适配（保留）
├── generic_adapter.go  # 标准 OpenRTB 2.5 适配（覆盖大多数 Exchange）
└── registry.go         # Exchange 注册表，按 ID 路由到对应适配器
```

大多数国内 SSP（Mintegral、穿山甲/Pangle、汇量等）支持标准 OpenRTB 2.5，`generic_adapter` 覆盖 80% 场景。仅协议有变体的 Exchange 需要单独 adapter。

### 2.2 花费安全护栏

| 护栏 | 机制 |
|------|------|
| 全局日预算上限 | 配置级别，所有 campaign 日花费总和不超过阈值（如 500 元/天） |
| 单次出价上限 | bidder 拒绝超过阈值的出价（如 CPM ≤ 50 元） |
| 余额告警 | 余额低于阈值时钉钉告警，低于最低值时自动暂停所有 campaign |
| 花费速率告警 | 5 分钟内花费超过预期速率时告警 |
| 一键熔断 | `POST /api/v1/admin/circuit-break` — 立即暂停所有竞价 |

### 2.3 数据对账

| 对账类型 | 频率 | 机制 |
|---------|------|------|
| 实时对账 | 每小时 | Redis 花费 vs ClickHouse 花费，差异 > 1% 告警 |
| 日终对账 | 每天 | 使用现有 `daily_reconciliation` 表，实现对账逻辑并自动运行 |
| Exchange 对账 | 每天 | 和 Exchange 侧花费报告比对，确认无计费偏差 |

### 2.4 autopilot 升级

- 模拟流量继续跑（作为基线对照）
- 真实流量数据纳入日报
- 新增对账结果监控
- 新增花费异常检测

## Phase 3：内测代理商入驻（第 3-5 月）

### 3.1 代理商入驻流程

```
代理商收到邀请链接
  → 填写公司信息
  → 你审批通过
  → 系统生成 API Key + 初始账号
  → 发送欢迎邮件（含登录凭证、操作指南）
```

内测阶段不开放公开注册。现有 `registration` 模块增加 `invite_code` 字段控制入口。

### 3.2 运营后台（Admin Dashboard）

路由 `/admin/*`，独立管理员 token 鉴权，和广告主 API Key 体系隔离。

| 功能 | 说明 |
|------|------|
| 代理商列表 | 所有代理商的余额、活跃 campaign 数、日花费 |
| 全局大盘 | 平台总花费、总请求量、胜率、收入（加价差） |
| 素材审核队列 | 现有 creative approval 补上批量审核 UI |
| 充值管理 | 内测阶段手动充值（线下转账 → 后台加余额） |
| 告警中心 | 汇总所有告警（花费异常、服务故障、对账差异） |

### 3.3 支付方案

内测阶段：线下对公转账 + 后台手动充值。充值记录保留完整审计日志。

在线支付（支付宝/微信支付）等合规资质到位后再接入，属于 Phase 3 后期或 Phase 4。

### 3.4 代理商自助功能补全

| 功能 | 现状 | 需要做的 |
|------|------|---------|
| 多 Campaign 管理 | 有 | 够用 |
| 花费报表导出 | 无 | CSV/Excel 导出，代理商需要给客户出报告 |
| 操作日志 | 无 | 记录谁在什么时候改了什么 |
| 子账号 | 无 | 暂不做，内测阶段一个代理商一个账号 |

### 3.5 autopilot 在 Phase 3 的角色

升级为运营监控工具：

- 每日对每个代理商生成投放健康报告
- 检测异常模式（点击欺诈突增、花费骤变）
- 代理商余额不足时提前告警运营方

## 合规路线（商务侧并行推进）

### 资质清单

| 资质 | 用途 | 预计周期 | 前置条件 |
|------|------|---------|---------|
| 营业执照 | 公司主体 | 如已有则跳过 | — |
| ICP 备案 | 网站合法运营 | 2-4 周 | 域名 + 服务器在国内 |
| ICP 经营许可证（B25） | 增值电信业务 | 2-3 月 | 注册资金 ≥ 100 万，3 人社保 |
| 互联网广告信息服务 | 广告经营合规 | 和 B25 一起申请 | 营业执照经营范围含广告 |
| PIPL 合规 | 用户数据保护 | 持续性工作 | 技术侧配合 |

### 技术侧合规配合

| 要求 | 技术实现 |
|------|---------|
| 广告标识 | 所有投放素材标注"广告"字样，bidder 出价时自动注入 |
| 广告审核留痕 | 素材审核记录不可删除，保存审核人、时间、结果 |
| 用户数据合规 | 设备 ID 脱敏存储，ClickHouse 中 device_id 做哈希处理 |
| 数据保留策略 | bid_log 6 个月 TTL（已有）；审计日志保留 3 年 |
| 隐私政策页面 | 前端新增 `/privacy` 页面 |
| 数据导出/删除 | PIPL 要求，预留 API |

### 时间线对齐

```
月份    技术侧                          商务侧
─────  ──────────────────────────────  ──────────────────────────
 1     生产部署 + autopilot verify      注册公司/补经营范围 + 买域名
 2     autopilot run + 生产加固         ICP 备案 + 启动 B25 申请
 3     Exchange 对接 + 安全护栏         谈 Exchange 合作
 4     小额实投 + 对账验证              跟进 B25 审批 + 联系代理商
 5     运营后台 + 代理商功能            B25 到手 + 代理商签约入驻
```

**关键路径：** B25 许可证是最大阻塞项。技术侧 Phase 1 和 Phase 2 不依赖它，可独立推进。Phase 3 代理商正式入驻需要资质到位。
