**Findings**
- High: 现在没有任何测试覆盖 `/bid/{exchange_id}` 这条 handler 链，因此完全漏掉了 exchange 路径和 direct 路径行为不一致的问题。`handlers` 集成测试虽然把 `ExchangeRegistry` 装进了 fixture [cmd/bidder/handlers_integration_test.go](</C:/Users/Roc/github/dsp/cmd/bidder/handlers_integration_test.go:76>) [cmd/bidder/handlers_integration_test.go](</C:/Users/Roc/github/dsp/cmd/bidder/handlers_integration_test.go:83>)，但文件里没有一条测试真正请求 `POST /bid/{exchange_id}`。这正好解释了为什么 exchange 路径漏了 click tracker 也没被测出来。
- High: `HandleStartCampaign` 的测试只覆盖了“余额足够且依赖全健康”的 happy path，没有覆盖最危险的 fail-closed 分支，给了这条链过高的安全感。[internal/handler/e2e_public_campaign_test.go](</C:/Users/Roc/github/dsp/internal/handler/e2e_public_campaign_test.go:83>) 的 `TestCampaign_StartPublishesActivated` 只断言了 `200 + activated pub/sub`，而支撑 harness 还特意把 advertiser 预充到足够余额 [internal/handler/e2e_support_test.go](</C:/Users/Roc/github/dsp/internal/handler/e2e_support_test.go:185>)。仓库里也没有对应的 `insufficient balance`、`GetBalance 出错`、`no creative`、`budget_total < budget_daily` 启动失败用例，所以当前那个“余额查询失败时仍然放行启动”的真实缺陷不会被任何测试抓到。
- Medium: 这套测试把“API 发布 activated”和“loader 收到 activated”分开测了，但没有一条跨模块测试证明 `POST /campaigns/{id}/start` 之后 bidder 真的能马上看到该 campaign 并参与竞价。当前只分别有 [internal/handler/e2e_public_campaign_test.go](</C:/Users/Roc/github/dsp/internal/handler/e2e_public_campaign_test.go:88>) 的 `TestCampaign_StartPublishesActivated` 和 [internal/bidder/loader_integration_test.go](</C:/Users/Roc/github/dsp/internal/bidder/loader_integration_test.go:112>) 的 `TestLoader_PubSubActivated`。这意味着 API 侧 action 名、payload shape、Redis channel 名只要有一处改坏，单边测试仍可能全绿。
- Medium: `TestHandlers_WinNormalCPM` 只校验 `/win` 的预算扣减和 Kafka 事件数量，没有校验事件 payload 本身，因此兜不住当前 win 事件字段漂移的问题。[cmd/bidder/handlers_integration_test.go](</C:/Users/Roc/github/dsp/cmd/bidder/handlers_integration_test.go:220>) 到 [cmd/bidder/handlers_integration_test.go](</C:/Users/Roc/github/dsp/cmd/bidder/handlers_integration_test.go:235>) 只数 `dsp.bids` 中有 1 条 win event，但不会检查 `creative_id`、`bid_price`、`clear_price`、`request_id` 是否与实际竞价结果一致。这样 `handleWin` 里“重算当前 bid price / 取第一个 creative”这类数据质量 bug 会直接漏过去。
- Medium: consumer 的 DLQ 集成测试只覆盖了非生产路径。生产里 `reporting.Store` 实现了 `BatchBidLogStore`，所以默认会走 `runBatchLoop` [cmd/consumer/runner.go](</C:/Users/Roc/github/dsp/cmd/consumer/runner.go:79>) [cmd/consumer/runner.go](</C:/Users/Roc/github/dsp/cmd/consumer/runner.go:123>)；但 [cmd/consumer/consumer_integration_test.go](</C:/Users/Roc/github/dsp/cmd/consumer/consumer_integration_test.go:202>) 的 `failingStore` 只实现 `InsertEvent`，`TestConsumer_CHFailureDLQ` 因而只验证了 single-row DLQ 路径 [cmd/consumer/consumer_integration_test.go](</C:/Users/Roc/github/dsp/cmd/consumer/consumer_integration_test.go:211>)。真正更贴近线上风险的“批量写 ClickHouse 失败时，整批 fan-out 到 DLQ”没有被集成测试覆盖。

**改进建议**
- 给 `cmd/bidder/handlers_integration_test.go` 增加一条 `POST /bid/{exchange_id}` 正向测试，至少断言 `NURL`、click tracking、响应格式和后续 `/click` 可达。
- 给 `internal/handler/e2e_public_campaign_test.go` 补 4 条启动失败测试：余额不足、余额查询错误、无 approved creative、`budget_total < budget_daily`。
- 新增一条跨模块链路测试：`POST /campaigns/{id}/start` -> Redis -> loader cache 更新 -> 紧接着发一个 bid 请求并命中该 campaign。
- 强化 `TestHandlers_WinNormalCPM`，把 Kafka 里的 win event 解出来断言 `campaign_id`、`request_id`、`clear_price`，并至少校验 `creative_id`/`bid_price` 不为默认错误值。
- 给 consumer 增加一个 batch-store failing fixture，专门验证 `runBatchLoop` 的 DLQ 行为，而不是只测 single insert 路径。

整体上，这些测试不是“写得差”，而是 happy path 很完整，但跨模块契约和失败路径还不够硬。当前最该补的是 exchange bid、campaign start 失败分支、以及 batch DLQ 这三块。
